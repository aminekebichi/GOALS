Working on the GOALS project with custom skills and MCP integration changed the way development felt compared to earlier assignments. Instead of treating Claude Code as just a tool to generate code snippets, it started to feel more like something we were configuring and guiding. The biggest shift came from reducing how much we had to think about process, and instead focusing more on what we actually wanted to build.

The custom /add-feature skill had the most noticeable impact on workflow. Before using it, adding a new feature meant mentally stepping through the same checklist every time. We would look at existing routes, figure out where logic should live, check what helper functions already existed, and then implement both backend and frontend pieces. None of that was particularly difficult, but it was repetitive and easy to miss small details. The skill essentially encoded that entire process into a reusable workflow.

For example, when implementing the /api/players/{id}/radar endpoint, the skill immediately pushed us to check existing helper functions before writing new logic. That led us directly to reuse the composite score functions instead of rebuilding them. Without that step being enforced, it would have been very easy to duplicate logic or introduce inconsistencies. Similarly, for the /api/teams/form endpoint, running the skill exposed a limitation where it did not explicitly tell us to check for local helper functions inside routers. We had to manually discover _load_fixtures() in calendar.py, which then led to improving the skill itself. That iteration from v1 to v2 made the workflow more complete by turning something we learned manually into something the system enforces automatically .

What stood out the most is that the skill reduced cognitive overhead. Instead of remembering all the project-specific constraints, like not modifying data files or maintaining consistent API response formats, those rules were built into the workflow. Over time, this made feature development more consistent and less error-prone. It also made it easier to pick up where we left off, since the process was no longer dependent on remembering details from previous sessions.

The MCP integration introduced a different kind of improvement. While the custom skill helped with writing code, MCP expanded what Claude Code could actually do. Without MCP, anything related to GitHub required leaving the development environment, opening a browser, and performing actions manually. With MCP, those tasks became part of the same workflow as coding.

A good example of this was creating issues for the remaining machine learning notebooks. Instead of manually writing three separate GitHub issues, Claude Code used the MCP server to list existing issues and then create new ones with structured titles and descriptions based on the project context . This was not just faster, it also ensured consistency because the issue descriptions were aligned with the specifications already defined in CLAUDE.md. The difference here is not just speed, but integration. MCP allowed the AI to operate across both the codebase and the project management layer, which is something that was not possible before.

This changes how development feels because it removes the boundary between writing code and managing a project. Instead of switching tools, everything happens in one place. It also opens up workflows that would otherwise be too tedious to automate manually, like generating issues, checking repository state, or eventually even reviewing pull requests.

Looking forward, there are a few areas where this system could be extended further. The most obvious next step is building more specialized skills. For example, a skill dedicated to implementing machine learning notebooks could take advantage of the existing constraints in CLAUDE.md and automate repetitive setup steps like ensuring proper train/test splits or applying consistent evaluation methods. Another useful skill would be something like a review or validation tool that checks code against project-specific rules before committing.

Hooks are another area that could improve the workflow. A pre-commit hook that checks for violations of core assumptions, such as accidentally shuffling temporal data, would help prevent subtle bugs from being introduced. Similarly, hooks could enforce rules around file access, such as preventing unintended writes to protected directories. These would act as guardrails that operate automatically without requiring explicit prompts.

Sub-agents also seem like a natural extension, especially for tasks that can be parallelized. The machine learning pipeline in GOALS involves multiple notebooks that are largely independent. Running each one in a separate sub-agent and then aggregating the results would significantly reduce total execution time and keep contexts cleaner. Another possible use would be a background agent that monitors long-running scraping processes and reports issues without interrupting the main workflow.

Overall, the combination of custom skills and MCP shifted development from a step-by-step manual process to something more structured and system-driven. The skill reduced the mental overhead of implementing features, while MCP expanded what tasks could be handled within a single environment. Together, they made the workflow more efficient and more consistent, while also highlighting how much of development is really about managing process rather than writing code.
